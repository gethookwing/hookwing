Agent experience

What it means to build for agents

Most infrastructure was designed for humans — dashboards, OAuth flows, browser confirmation steps. Hookwing is designed for agents. Six principles define what that means in practice.

Start free Read the docs

An agent can't solve a CAPTCHA. It can't click through an OAuth consent screen. It doesn't have a browser session, and it doesn't want one.

Infrastructure built for humans puts agents in a bind: authentication requires a dashboard, provisioning requires a human to configure it first, and errors are written for a developer to read in a terminal rather than a system to parse and act on.

Agent experience (AX) is the set of design decisions that determine whether a system works unattended. Hookwing is built around six of them.

Six AX principles

Principle 1

API-key auth by default

Agents authenticate with a key. No OAuth redirect, no browser session, no human confirming a consent screen. The first API call works the same way as the thousandth.

Every Hookwing endpoint accepts a bearer token: Authorization: Bearer <key>. Keys are scoped (read-only, write, admin) and provisioned via API — no dashboard required.
Principle 2

Self-provisioning

An agent creates its own webhook endpoint. No human needs to log in to a dashboard first. The agent makes an API call, receives a URL and a secret, and starts receiving events.

POST /v1/endpoints returns a stable URL and an HMAC secret in a single response. The agent owns that endpoint. It can update delivery rules, pause ingestion, and delete the endpoint when the task is done.
Principle 3

Machine-readable everything

Pricing, errors, status, rate limits — all available as structured JSON. No scraping, no parsing HTML, no guessing what a response means.

GET /v1/pricing returns current plan limits and overage costs. Error responses follow a consistent schema: { "code": "endpoint_not_found", "message": "...", "docs_url": "..." }. The status page at /v1/status returns component health as a typed object.
Principle 4

Structured errors agents can act on

Error responses include a machine-readable code, a human-readable message, and a documentation URL. An agent can branch on the code without parsing prose.

Every 4xx and 5xx response includes "code" (stable, lowercase, underscore-separated), "message" (plain English), and "docs_url" (link to the relevant reference). No ambiguous HTTP status codes alone. No error messages written only for humans.
Principle 5

MCP-native

Hookwing ships an MCP server. Agents that support the Model Context Protocol can provision endpoints, inspect delivery logs, and manage their webhook infrastructure as native tool calls — no custom integration required.

The Hookwing MCP server exposes create_endpoint, list_events, get_delivery_status, retry_event, and delete_endpoint as typed tools. Any MCP-compatible runtime can use them directly. Read more: MCP integration docs.
Principle 6

No CAPTCHAs by default

Hookwing does not require CAPTCHA solving at any point in the agent workflow. Signup, provisioning, delivery — all accessible via API key. No human verification step in the critical path.

Rate limiting and abuse prevention are handled at the infrastructure layer. The API enforces per-key quotas and detects anomalous patterns without requiring a human to prove they are human.

AX at a glance

  • API-key auth — no OAuth, no browser
  • Self-provisioning via POST /v1/endpoints
  • Machine-readable pricing at GET /v1/pricing
  • Typed error codes agents can branch on
  • MCP server with five native tools
  • No CAPTCHAs in any API flow
