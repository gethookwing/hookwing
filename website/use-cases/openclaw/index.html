The platform

What is OpenClaw?

OpenClaw is an AI agent orchestration platform. It runs autonomous agents — coding agents, content agents, DevOps agents — that communicate over channels like Discord, Slack, and Telegram. Agents spawn sub-agents, manage workspaces, and operate 24/7 without human intervention.

The challenge

Agents need events. Polling doesn't scale.

AI agents need to react to external events — a PR opened on GitHub, a payment on Stripe, an alert from your monitoring stack. The naive approach is polling, but polling is wasteful, adds latency, and breaks when agents restart.

⏱️

Latency

Polling every 30 seconds means up to 30 seconds of wasted time before an agent reacts. Webhooks deliver in milliseconds.

🔄

Reliability

If an agent restarts during a poll cycle, events get missed. Hookwing holds events and retries delivery until the agent is back.

📡

Fan-out

One event often needs to reach multiple agents. Building pub/sub from scratch is complex. Hookwing does fan-out natively.

Integration patterns

How OpenClaw agents use Hookwing

These are real patterns running in production. OpenClaw agents use Hookwing as their event backbone.

GitHub PR opened → coding agent reviews it

GitHub fires a pull_request.opened webhook to Hookwing. Hookwing verifies the signature and delivers it to the coding agent's endpoint. The agent reads the diff, posts a review, and suggests fixes — all without polling GitHub's API.

GitHub Hookwing Coding agent reviews PR

Agent A finishes work → Agent B picks it up

OpenClaw runs specialized agents — one for code, one for content, one for DevOps. When the coding agent finishes a feature, it POSTs the result to Hookwing. The content agent receives it and updates the docs. No shared queue, no direct networking.

Coding agent Hookwing Content agent

Payment succeeds → agent provisions the account

Stripe sends payment_intent.succeeded to Hookwing. Hookwing fans it out: one delivery to the provisioning agent, one to the CRM agent, one to analytics. Each retries independently. If the provisioning agent is mid-deploy when the payment lands, Hookwing holds the event.

Stripe Hookwing Provisioning + CRM + Analytics

Alert fires → DevOps agent investigates

An uptime monitor detects a service is down and sends a webhook to Hookwing. The DevOps agent receives it, checks logs, identifies the root cause, and either fixes it automatically or escalates to a human with full context. No manual triage step.

Monitor alert Hookwing DevOps agent investigates

Agent polls Hookwing when ready — no public endpoint needed

Not every agent has a public URL. OpenClaw agents behind NAT or on local machines can poll the Hookwing events API. Events are stored durably — the agent checks when it wakes up and processes everything it missed. Push or pull, same reliability.

Implementation

Set up in minutes, not days

An OpenClaw agent can self-provision its own webhook infrastructure via the Hookwing API. No dashboard clicks, no manual setup.

1. Agent creates its own endpoint

curl -X POST https://api.hookwing.com/v1/endpoints \
  -H "Authorization: Bearer $HOOKWING_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://agent.internal/webhooks",
    "name": "github-pr-listener",
    "eventTypes": ["pull_request.*"]
  }'

2. Configure the source (e.g. GitHub) to send to Hookwing

# GitHub sends webhooks to this ingest URL:
https://api.hookwing.com/v1/ingest/ep_your_endpoint_id

# Hookwing accepts, stores, and delivers to your agent.
# If your agent is down, Hookwing retries up to 6 times over ~8.5 hours.

3. Agent verifies and processes the webhook

import { Webhook } from '@hookwing/sdk';

const wh = new Webhook(process.env.HOOKWING_SIGNING_SECRET);

app.post('/webhooks', async (req, res) => {
  const event = await wh.verify(req.body, {
    signature: req.headers['x-hookwing-signature'],
    timestamp: req.headers['x-hookwing-timestamp'],
  });

  // Route to the right agent handler
  if (event.type === 'pull_request.opened') {
    await reviewPR(event.payload);
  }

  res.status(200).send('OK');
});

4. Or pull mode — no public endpoint needed

# Agent behind NAT? Just poll for events when ready:
curl "https://api.hookwing.com/v1/events?since=1774000000000&limit=50" \
  -H "Authorization: Bearer $HOOKWING_API_KEY"
Why Hookwing

Built for agents, not just humans

Most webhook platforms assume a human is configuring everything through a dashboard. Hookwing is API-first — agents can self-provision, manage keys, and inspect events programmatically.

API-first

Every operation available via REST API. Agents create endpoints, manage keys, replay events, and check analytics — no browser required.

MCP server

Install the Hookwing MCP server to give Claude Code, Cursor, or any MCP-compatible agent full webhook management capabilities.

Self-provisioning

Agents sign up, create endpoints, and generate scoped API keys. No human in the loop. No CAPTCHAs.

Automatic retries

6 retry attempts over ~8.5 hours with exponential backoff. If your agent restarts, the event is waiting when it comes back.

Event routing

Route events by type, payload content, or headers. Send order.* events to the billing agent and user.* events to the CRM agent.

Machine-readable pricing

Agents can query /api/pricing for JSON tier data. No scraping. Enables autonomous upgrade decisions.
