Event routing

Updated 2026-03-26
✦ Available on Warbird and above

Send the right event to the right endpoint, automatically. Routing rules evaluate each inbound event against conditions you define, then deliver, drop, or transform accordingly — with no code changes on your receiver.

By default, Hookwing delivers every event to the endpoint that received it, and fan-out copies it to any other endpoint with fan-out enabled. Event routing is additive: rules create additional conditional deliveries on top of that baseline. A rule matches an event, then delivers it to whichever endpoint you specify — regardless of where the event originally landed.

Common uses:

  • Route payment.completed events to a billing endpoint and suppress everything else
  • Fan-out high-value orders (amount > 1000) to a separate priority endpoint
  • Send events from a specific source header to a dedicated processing pipeline
  • Drop internal test events before they hit your production handlers
  • Extract a subset of payload fields before delivery to reduce downstream payload size

How rules work

Each routing rule has three parts: conditions, an action, and an optional transform.

  1. Conditions — one or more field checks that must all pass (AND logic). If all conditions match, the rule fires.
  2. Action — what to do when the rule matches: deliver to an endpoint, or drop the event.
  3. Transform — an optional payload transformation applied before delivery.

Rules are evaluated in priority order (lowest number first) for every inbound event. All matching rules fire — there is no early exit after the first match. A drop action suppresses that rule's delivery but does not prevent other rules from matching.

Conditions

A condition checks a single field against a value using an operator. Every condition in a rule must match for the rule to fire.

{
  "field": "event.type",
  "operator": "equals",
  "value": "order.created"
}

Multiple conditions on one rule:

[
  { "field": "event.type",        "operator": "equals", "value": "order.created" },
  { "field": "$.payload.amount",  "operator": "gt",     "value": 500 }
]

Both must match. There is no OR logic within a single rule — create separate rules with the same action if you need OR behaviour.

Operators

Operator Description Value type Example
equals Strict equality string, number, boolean "value": "order.created"
not_equals Field is not equal to value string, number, boolean "value": "test"
contains Field string contains value as substring string "value": "@example.com"
starts_with Field string starts with value string "value": "order."
gt Field (numeric) is greater than value number "value": 100
gte Field (numeric) is greater than or equal to value number "value": 100
lt Field (numeric) is less than value number "value": 50
lte Field (numeric) is less than or equal to value number "value": 50
exists Field is present (true) or absent (false) boolean "value": true
in Field value is one of the listed values array "value": ["order.created", "order.updated"]
regex Field string matches the regular expression string (RegExp) "value": "^order\\."

Field selectors

The field property uses a selector syntax to address parts of the event:

Selector What it addresses Example
event.type The event type string "field": "event.type"
$.payload.<path> A field inside the event payload, dot-separated for nesting "field": "$.payload.amount", "field": "$.payload.user.role"
headers.<name> An HTTP header on the inbound request "field": "headers.X-Source"
$.headers.<name> Same as above, alternative syntax "field": "$.headers.X-Source"

Actions

The action object defines what happens when a rule matches.

FieldRequiredDescription
type required "deliver" or "drop". deliver sends the event to endpointId. drop suppresses this rule's delivery — does not prevent other rules from matching.
endpointId optional Required when type is "deliver". The target endpoint ID. Must belong to your workspace.
transform optional A transform to apply to the payload before delivery. See Transforms.

Transforms

Transforms modify the event payload before it reaches the target endpoint. Available on Warbird (extract only) and Stealth Jet (all transforms).

extract

Returns only the listed top-level fields from the payload. Useful for reducing payload size or limiting what a downstream service receives.

{
  "type": "extract",
  "config": {
    "fields": ["order_id", "amount", "status"]
  }
}

Input: { "order_id": "ord_123", "amount": 150, "status": "paid", "customer_email": "..." }
Output: { "order_id": "ord_123", "amount": 150, "status": "paid" }

rename

Renames top-level fields according to a mapping. Useful when a downstream service expects different field names. Available on Stealth Jet.

{
  "type": "rename",
  "config": {
    "mapping": {
      "order_id": "id",
      "customer_email": "email"
    }
  }
}

template

Renders a string template using {{field}} interpolation. Produces a string output rather than an object — useful for notification payloads or simple triggers. Available on Stealth Jet.

{
  "type": "template",
  "config": {
    "template": "Order {{order_id}} — {{amount}} paid"
  }
}

Output: "Order ord_123 — 150 paid"

Missing fields render as empty string. Template transforms are applied to the top-level payload object only.

Priority and evaluation order

Each rule has a priority field (integer, 0–1000). Rules are evaluated in ascending order — lower number runs first. All matching rules fire; there is no stop-on-first-match.

Priority affects evaluation order only. Two rules with the same priority are evaluated in creation order.

Disabled rules ("enabled": false) are skipped entirely.

API reference

Base URL: https://api.hookwing.com
All endpoints require Authorization: Bearer <api_key>.
Routing rule endpoints require the endpoints:read or endpoints:write scope as noted.

POST /v1/routing-rules

Create a new routing rule. Requires endpoints:write scope.

Request body

FieldTypeDescription
namerequiredstringHuman-readable name. Max 100 characters.
conditionsrequiredarrayOne or more condition objects. All must match (AND). Minimum 1.
actionrequiredobjectThe action to take when conditions match.
priorityoptionalintegerEvaluation order, 0–1000. Default: 0.
enabledoptionalbooleanWhether the rule is active. Default: true.

Example request

curl -X POST https://api.hookwing.com/v1/routing-rules \
  -H "Authorization: Bearer hw_live_..." \
  -H "Content-Type: application/json" \
  -d '{
    "name": "High-value orders to priority endpoint",
    "priority": 10,
    "conditions": [
      { "field": "event.type",       "operator": "equals", "value": "order.created" },
      { "field": "$.payload.amount", "operator": "gt",     "value": 500 }
    ],
    "action": {
      "type": "deliver",
      "endpointId": "ep_live_abc123"
    }
  }'

Response 201 Created

{
  "id": "rule_01jq...",
  "workspaceId": "ws_...",
  "name": "High-value orders to priority endpoint",
  "priority": 10,
  "conditions": [
    { "field": "event.type",       "operator": "equals", "value": "order.created" },
    { "field": "$.payload.amount", "operator": "gt",     "value": 500 }
  ],
  "action": {
    "type": "deliver",
    "endpointId": "ep_live_abc123",
    "transform": null
  },
  "enabled": true,
  "createdAt": 1774000000000,
  "updatedAt": 1774000000000
}

Error responses

  • 400 — Invalid input (missing required fields, unknown operator, empty conditions array)
  • 403 — Routing rules not available on your tier, or rule limit reached
  • 404 — endpointId not found or not in your workspace

GET /v1/routing-rules

List all routing rules for your workspace, ordered by priority ascending. Requires endpoints:read scope.

Example request

curl https://api.hookwing.com/v1/routing-rules \
  -H "Authorization: Bearer hw_live_..."

Response 200 OK

{
  "rules": [
    {
      "id": "rule_01jq...",
      "name": "High-value orders to priority endpoint",
      "priority": 10,
      "conditions": [...],
      "action": { "type": "deliver", "endpointId": "ep_live_abc123", "transform": null },
      "enabled": true,
      "createdAt": 1774000000000,
      "updatedAt": 1774000000000
    }
  ]
}

GET /v1/routing-rules/:id

Retrieve a single routing rule by ID. Requires endpoints:read scope.

Example request

curl https://api.hookwing.com/v1/routing-rules/rule_01jq... \
  -H "Authorization: Bearer hw_live_..."

Error responses

  • 404 — Rule not found or not in your workspace

PATCH /v1/routing-rules/:id

Update a routing rule. All fields are optional — only the fields you send are updated. Requires endpoints:write scope.

Updatable fields

FieldTypeDescription
namestringNew name. Max 100 characters.
conditionsarrayReplace the condition set. Minimum 1.
actionobjectReplace the full action object.
priorityintegerNew priority, 0–1000.
enabledbooleanEnable or disable the rule without deleting it.

Example — disable a rule

curl -X PATCH https://api.hookwing.com/v1/routing-rules/rule_01jq... \
  -H "Authorization: Bearer hw_live_..." \
  -H "Content-Type: application/json" \
  -d '{ "enabled": false }'

Error responses

  • 400 — Invalid input
  • 403 — Transform type not available on your tier
  • 404 — Rule or endpoint not found

DELETE /v1/routing-rules/:id

Permanently delete a routing rule. Requires endpoints:write scope. Returns 204 No Content.

Example request

curl -X DELETE https://api.hookwing.com/v1/routing-rules/rule_01jq... \
  -H "Authorization: Bearer hw_live_..."

POST /v1/routing-rules/test

Dry-run an event against your rules without creating any deliveries. Requires endpoints:read scope. Useful for verifying rule logic before enabling.

Request body

FieldDescription
event.typerequiredThe event type to test against.
event.payloadoptionalPayload object to match payload conditions.
event.headersoptionalHeaders object to match header conditions.
ruleIdoptionalTest against a specific rule only. Omit to test against all rules.

Example request

curl -X POST https://api.hookwing.com/v1/routing-rules/test \
  -H "Authorization: Bearer hw_live_..." \
  -H "Content-Type: application/json" \
  -d '{
    "event": {
      "type": "order.created",
      "payload": { "amount": 750, "status": "paid" },
      "headers": { "X-Source": "stripe" }
    }
  }'

Response 200 OK

{
  "matched": true,
  "matchedRules": [
    {
      "ruleId": "rule_01jq...",
      "name": "High-value orders to priority endpoint",
      "action": "deliver",
      "transform": null
    }
  ],
  "transforms": null
}

When matched is false, matchedRules is an empty array. No deliveries are created regardless of the result.

Examples

Route by event type

Deliver all payment events to a dedicated payments endpoint, regardless of which endpoint originally received them.

{
  "name": "All payment events to payments endpoint",
  "conditions": [
    { "field": "event.type", "operator": "starts_with", "value": "payment." }
  ],
  "action": {
    "type": "deliver",
    "endpointId": "ep_live_payments"
  }
}

Route multiple types with a single rule

Use the in operator to match a list of event types without creating one rule per type.

{
  "name": "User lifecycle events to CRM",
  "conditions": [
    {
      "field": "event.type",
      "operator": "in",
      "value": ["user.created", "user.updated", "user.deleted"]
    }
  ],
  "action": {
    "type": "deliver",
    "endpointId": "ep_live_crm"
  }
}

Route by payload field value

High-value orders (amount > 500) go to a priority processing endpoint.

{
  "name": "High-value orders",
  "priority": 5,
  "conditions": [
    { "field": "event.type",       "operator": "equals", "value": "order.created" },
    { "field": "$.payload.amount", "operator": "gt",     "value": 500 }
  ],
  "action": {
    "type": "deliver",
    "endpointId": "ep_live_priority"
  }
}

Route by inbound header

Events from a specific source (identified by custom header) go to a separate endpoint.

{
  "name": "Stripe events to Stripe processor",
  "conditions": [
    { "field": "headers.X-Source", "operator": "equals", "value": "stripe" }
  ],
  "action": {
    "type": "deliver",
    "endpointId": "ep_live_stripe"
  }
}

Drop test events

Suppress events from test environments before they reach any endpoint. Drop rules still allow other matching rules to fire.

{
  "name": "Drop test environment events",
  "priority": 1,
  "conditions": [
    { "field": "$.payload.environment", "operator": "equals", "value": "test" }
  ],
  "action": { "type": "drop" }
}

Extract fields before delivery

Strip sensitive fields from the payload before delivering to a third-party endpoint. Requires Warbird or above.

{
  "name": "Order summary to analytics",
  "conditions": [
    { "field": "event.type", "operator": "equals", "value": "order.created" }
  ],
  "action": {
    "type": "deliver",
    "endpointId": "ep_live_analytics",
    "transform": {
      "type": "extract",
      "config": { "fields": ["order_id", "amount", "currency", "status"] }
    }
  }
}

Check for presence of a field

Route events that include a refund_id field to a refunds handler.

{
  "name": "Refund events to refund handler",
  "conditions": [
    { "field": "$.payload.refund_id", "operator": "exists", "value": true }
  ],
  "action": {
    "type": "deliver",
    "endpointId": "ep_live_refunds"
  }
}
